# Output Rules

## 톤과 문말

- 존댓말을 쓰면 `했습니다`, `이었습니다` 같은 문말을 일관되게 유지한다.
- 겸손하되 자신감 있는 톤을 유지하고, 과장이나 허위 없이 구체적 경험 중심으로 쓴다.
- 회사와 직무에 맞는 어휘를 사용하되 공고 문구를 그대로 복붙하지 않는다.

## 글자 수

- 문항별 제한이 있으면 반드시 지킨다.
- 각 `qN.md`에는 `## 글자 수 제한` 섹션을 두고 문항별 기준을 명시한다.
- `scripts/validation.py`는 이 섹션을 기준으로 최종 답변 길이를 검사한다.
- 초과하면 반복 설명, 부사, 일반론부터 줄인다.
- 제한이 없으면 기본적으로 500자에서 800자 안쪽의 가독성을 우선한다.

## 금기와 피할 표현

- `저는 ~한 사람입니다` 같은 자기 규정 문장을 반복하지 않는다.
- `~할 수 있다고 생각합니다` 같은 모호한 표현은 줄인다.
- `기준`이라는 단어는 쓰지 않는다.
- `얼마나 중요한지 몸으로 익혔습니다` 같은 과하게 정리된 학습형 문장은 쓰지 않는다.
- `구조`, `구조적`, `구조적으로` 같은 메타 표현은 쓰지 않는다.
- `~보다`, `~보다 먼저` 같은 비교 표현으로 우선순위를 설명하지 않는다.
- `화려한 기능`, `빠른 기능 구현`처럼 맥락 없는 허수 비교 표현은 쓰지 않는다.
- `영향도가 큰 항목`, `핵심 역량`, `효율적인 커뮤니케이션`처럼 장면이 보이지 않는 추상 표현은 피한다.
- `저는 단순히 ~로 보지 않고` 같은 상투적인 대비 문구도 피한다.
- 회사와 직무와 무관한 일반론은 쓰지 않는다.
- 청자가 모를 가능성이 큰 프로젝트명, 스터디명, 서비스명 같은 고유명사는 그대로 쓰지 않는다.
- 다만 회사명처럼 인지도가 있고 경력 메리트가 분명한 고유명사는 그대로 쓸 수 있다.
- 필요하면 설명형 표현으로 풀어 쓴다.
- 쉼표는 꼭 필요한 경우에만 최소한으로 쓴다.
- 소괄호는 쓰지 않는다.
- 작은따옴표와 큰따옴표는 쓰지 않는다.
- 가운데점은 쓰지 않는다.
- 위 기호 대신 문장을 나눠 자연스럽게 연결한다.

## 형식

- 문항 구분이 있으면 문항 번호나 제목을 분명히 표시한다.
- CLI에서 문항 답변을 보여줄 때는 항상 질문 원문, 현재 공백 포함 글자 수, 문항 글자 수 제한을 함께 표시한다.
- 사용자가 작성 과정을 보고 싶어 하면 `초안`, `리뷰어 평가`, `수정 반영`, `최종 답변`을 순서대로 함께 보여준다.
- `리뷰어 평가`는 `+`와 `-` 나열형보다 `리뷰어 총평`, `핵심 보완 포인트`, `최종 판정` 중심으로 정리한다.
- `수정 반영`에는 reviewer 지적 중 무엇을 어떻게 바꿨는지 문장 수준으로 짧게 적는다.
- `## 참조 자산`에는 실제로 읽은 파일 경로를 `- 경로 | 활용 메모` 형식으로 남긴다.
- `## 참조 자산`에서는 공통 과거 지원서보다 현재 회사 폴더의 기존 질문 파일과 before, after, comparison 파일을 우선 적는다.
- `## 질문 태그`에는 `대표 질문 유형:`과 `보조 태그:`를 적는다.
- `대표 질문 유형:`은 `question-library/`의 폴더 이름 체계를 그대로 사용한다.
- `## 유사 문항 참조`에는 `companies/*/questions/*.md` 경로를 `- 경로 | 유사한 질문 의도와 가져올 포인트` 형식으로 남긴다.
- 유사 문항은 `대표 질문 유형`이 먼저 맞는지 보고 고른다.
- 유사 문항이 있으면 합격 자소서보다 먼저 내 기존 문항의 경험 배치와 문장 흐름을 검토한다.
- 단락은 1문장에서 2문장 단위로 읽기 쉽게 끊는다.
- 문단은 결론을 먼저 두고 근거와 사례를 뒤에 붙이는 연역형으로 쓴다.
- 각 문단의 첫 줄에는 `[문단 요약]` 형식의 짧은 한 줄 요약을 먼저 둔다.
- 요약 문장은 한 문장으로 쓰고 15자에서 35자 안쪽을 기본값으로 잡는다.
- `[문단 요약]` 안에는 마침표를 쓰지 않는다.

## 적용 시점

- 초안 작성부터 수정과 첨삭 단계까지 모두 이 규칙을 유지한다.
