# Improvement Notes

## 2026-03-11 Koscom 실전 자소서 작업 메모

### few-shot 보완 필요

- 갈등 해결 문항 예시 추가
  노하우 -> 갈등 장면 -> 해결 기준 -> 결과와 배운점 구조 예시가 필요하다.
- 기술적 갈등 예시 추가
  선호 기술 충돌 자체보다 문제 정의를 다시 맞추고 요구사항 기준으로 합의하는 흐름 예시가 필요하다.
- 강점과 약점 문항 예시 추가
  기술 경험을 그대로 자랑하는 방식이 아니라 일하는 방식의 강점으로 번역하는 예시가 필요하다.
- 지원동기 문항 예시 보강
  회사 설명은 사실형으로 쓰고, 지원 이유와 해석은 지원자 언어로 쓰는 대비 예시가 필요하다.

### 작성 스타일 보완 필요

- 경험을 나열형으로 쓰지 말고 장면감 있게 풀어내는 예시가 더 필요하다.
- 경험 서술이 건조해지지 않도록 실제로, 당시, 처음에는, 이후 같은 전환 표현을 적절히 써서 흐름을 살리는 연습이 필요하다.
- 기술 경험도 설명문처럼 쓰기보다 문제 상황과 판단, 전환점을 더 선명하게 드러내야 한다.

### 모델 작업 메모

- 지원동기 문항에서 회사 설명을 해석형으로 쓰는 실수가 있었다.
- 강점과 약점 문항에서 기술 성과 중심으로 과도하게 밀어붙이는 경향이 있었다.
- 갈등 문항에서 관점 차이와 실제 갈등을 더 엄격히 구분해야 한다.

## 2026-03-13 IBK기업은행 Q1 사용자 확정본 반영 메모

### 표현 선택에서 배운 점

- 지원동기 문항의 기준 문장은 `생각했습니다`보다 `생각합니다`처럼 현재도 유효한 판단 기준으로 쓰는 편이 힘이 더 좋다.
- `작은 오류 하나`, `무엇보다 중요하다`처럼 체감 강도를 높이는 단어 선택이 문장 밀도를 끌어올린다.
- 비교 표현은 무조건 피할 것이 아니라 `기능 구현보다 먼저 정합성과 신뢰`처럼 실제 우선순위를 드러내는 경우에는 오히려 기준을 또렷하게 만든다.
- `신뢰 가능한`보다 `신뢰할 수 있는`처럼 덜 번역투인 표현이 제출 문안 톤에 더 잘 맞는다.
- 다만 `기준`, `가치`, `태도` 같은 메타 단어가 반복되면 설명문처럼 읽힐 수 있으므로 실제 오류 해결 과정과 운영 경험을 더 먼저 보여줘야 한다.
- `영향도가 큰 항목`처럼 작업 장면이 바로 떠오르지 않는 추상 표현은 피하고 `고객사가 먼저 확인해야 하는 문의`, `오류가 반복되는 언어와 표현`처럼 실제 단위로 내려 써야 한다.
- `저는 단순히 ~로 보지 않고` 같은 문장은 흔한 대비 문구라 힘이 약하므로, 무엇을 관찰했고 어떤 가설을 세웠는지 바로 쓰는 편이 낫다.

### 승격 후보

- 지원동기 문항 few-shot 보강 시 `허수 비교는 제거하고 실제 우선순위를 드러내는 비교는 살린다`는 예시를 추가할 가치가 있다.
