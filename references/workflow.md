# Workflow

## 기본 원칙

이 스킬은 회사별 작업공간을 만들고, 각 문항을 완성도 높은 결과물로 만드는 것을 목표로 한다. 필요한 질문만 하고, 사용자가 별도 지시를 주지 않으면 가능한 범위에서 스스로 판단해 진행한다.

## 시작 절차

초안 작성 전에 반드시 아래 순서를 지킨다. 순서가 비어 있으면 다음 단계로 넘어가지 않는다.

### 입력 수집 순서

1. 지원 회사명과 직무명을 받는다.
2. 채용 공고 원본 링크를 받는다.
3. 공고를 `company.md`에 저장하고 회사 이해, 직무 이해, 공고 핵심 키워드, 자소서 반영 포인트까지 먼저 정리한다.
4. 회사별 합격 자소서와 참고자료를 받는다.
5. 자소서 문항 전체와 글자 수 제한을 한 번에 받는다.
6. 공통 자산과 회사 자료를 읽고 문항별 경험 배치안을 만든다.
7. 경험 배치안을 사용자와 합의한다.
8. 그다음부터 현재 문항 하나씩 초안을 작성한다.

### 초안 착수 전 게이트

- `company.md`의 회사 이해와 직무 이해가 비어 있으면 초안을 쓰지 않는다.
- 합격 자소서나 회사 참고자료를 아직 받지 못했다면, 먼저 자료 요청이나 보조 레퍼런스 확보를 우선한다.
- 문항 전체를 받기 전에는 개별 문항 초안을 쓰지 않는다.
- 문항별 경험 배치안을 사용자와 먼저 맞추지 않았다면 초안을 쓰지 않는다.

1. 지원 회사와 직무를 먼저 확인한다.
2. `companies/<company-name>/` 폴더가 없으면 생성한다.
3. 채용 공고는 가능한 한 원본 링크로 받는다.
4. 원본 공고 링크는 `scripts/import_job_posting.py <url> companies/<company-name>` 형태로 파싱해 `companies/<company-name>/company.md`에 저장한다.
5. 링크 기반 파싱이 불가능한 경우에만 공고 원문이나 JD를 직접 받아 `companies/<company-name>/company.md` 기준 정보로 사용한다.
6. `companies/<company-name>/company.md`에는 공고 원문만 저장하지 말고, 아래 해석 섹션을 함께 정리한다.
   - 회사 이해
   - 직무 이해
   - 공고 핵심 키워드
   - 자소서 반영 포인트
7. 회사 이해에는 회사가 맡는 역할, 산업 내 위치, 왜 이 회사가 중요한지를 짧게 정리한다.
8. 직무 이해에는 실제 담당 업무, 요구 역량, 어떤 문제를 해결하는 역할인지 정리한다.
9. 공고 핵심 키워드에는 반복적으로 보이는 기술, 태도, 평가 포인트를 추린다.
10. 자소서 반영 포인트에는 지원동기, 직무 적합성, 강점 문항에 어떻게 연결할지 메모한다.
11. 합격 자소서 링크, 인재상, 기업문화, 면접 후기, 현직자 글, 블로그 글, 자소서 팁 같은 회사별 참고자료가 있으면 받는다.
12. 합격 자소서 링크가 여러 개면 한 번에 모아 받는다.
13. 링크로 받은 합격 자소서는 `scripts/import_linkareer_reference.py <url1> <url2> ... companies/<company-name>` 형태로 파싱해 `companies/<company-name>/references/` 아래에 개별 파일로 저장한다.
14. 링크가 아닌 참고자료도 `companies/<company-name>/references/` 아래에 별도 파일로 정리한다.
15. 합격 자소서가 아니더라도 해당 회사를 이해하는 데 도움이 되는 자료라면 `companies/<company-name>/references/`에 함께 보관하고 작성 전에 읽는다.
16. 현재 회사 자료가 부족하거나 문항이 매우 유사한 경우에는 다른 회사의 합격 자소서도 보조 레퍼런스로 받을 수 있다.
17. 타회사 합격 자소서는 문항 대응 방식, 문단 구조, 전개 흐름을 참고하는 용도로만 쓰고 현재 회사 자료보다 낮은 우선순위로 읽는다.
18. 기업과 무관하게 수집하는 합격 자소서 링크는 `question-library/`에 질문 유형별로 정리한다.
19. `question-library/`의 각 항목에는 질문 원문, 답변 원문, 대표 질문 유형, 보조 태그, 구조 분석을 함께 남긴다.
20. 새 레퍼런스에서 좋은 구조를 발견해도 바로 공통 규칙으로 올리지 말고 먼저 `question-library/`에 누적한다.
21. 같은 구조나 전략이 여러 사례에서 반복 확인되면 `references/few-shot.md`에 일반화된 예시로 승격한다.
22. `few-shot`에서도 반복적으로 유효하다고 판단되는 패턴만 `references/writing.md`, `references/workflow.md`, `references/evaluation.md` 같은 공통 규칙 문서에 반영한다.
23. 이 갱신 순서는 항상 `question-library -> few-shot -> rules`를 따른다.
24. 사용자가 `company.md` 최종본이나 이에 준하는 확정 해석을 주면, 해당 `companies/<company-name>/company.md`를 이후 유사 질문에서도 우선 반영하는 기준 자산으로 취급한다.
25. 이때 재사용 대상은 질문 의도 해석, 경험 연결 방식, 강조 기준, 피할 해석 같은 사용자 고유 판단 기준이다.
26. 회사 고유 사실관계와 공고 원문도 계속 `companies/<company-name>/company.md`를 기준으로 관리한다.
27. 자소서 문항 전체와 글자 수 제한을 먼저 받는다.
28. 문항 전체를 받기 전에는 어떤 경우에도 개별 문항 초안을 먼저 쓰지 않는다.
29. 공통 자산과 회사 자료를 읽고 어떤 경험을 어떤 문항에 배치할지 먼저 정리한다.
30. 한 문항에 경험이 과하게 몰리지 않도록 조정하고, 가장 강한 경험을 가장 적합한 문항에 우선 배치한다.
31. 실제 작성에 들어가기 전에 문항별 경험 배치안을 CLI에 먼저 보여주고 사용자와 합의한다.
32. 경험 배치안에는 최소한 `q번호`, `문항 핵심 의도`, `사용할 핵심 경험`, `보조 경험 여부`, `배치 이유`가 들어가야 한다.
33. 사용자가 배치안을 승인하거나 수정하면 그다음부터 실제 작성은 현재 문항 하나씩 진행한다.
34. 작성 전에 현재 문항에 어떤 경험을 쓸지 다시 확인하고, 필요하면 상황, 역할, 행동, 결과와 함께 해석까지 구조화한다.
35. 첫 초안의 문단 구조가 흔들리거나 리라이트 방향이 애매하면 `references/few-shot.md`의 예시와 `question-library/`의 같은 질문 유형 예시를 먼저 읽고 형식을 맞춘다.
36. 사용자가 초안에 대한 피드백만 주는 경우뿐 아니라, 완성에 가까운 문안을 직접 주는 경우에도 그 버전을 우선 반영한다.
37. 사용자가 직접 고친 문안이 들어오면 가능하면 기존 초안과 수정본을 각각 `before`, `after` 형태로 저장한다.
38. `before`와 `after`를 비교해 무엇이 좋아졌는지 구조, 문체, 경험 배치, 해석 방식 기준으로 분석 메모를 남긴다.
39. 이런 비교에서 같은 개선 패턴이 반복되면 먼저 `question-library/` 또는 작업 메모에 누적하고, 다시 반복 확인되면 `references/few-shot.md`와 공통 규칙 문서로 승격한다.

## 공통 자산 참조

작성 전에 다음 자산을 우선 참조한다.

1. 사용자가 확정한 `companies/<company-name>/company.md`
2. `assets/resume.md`
3. `assets/portfolio.md`
4. `assets/credentials.md`
5. `assets/past-applications/` 내 관련 문서
6. `companies/<company-name>/references/` 내 관련 문서

사용자가 특정 자산이나 경험을 우선하라고 지정하면 그 지시를 따른다. 별도 지시가 없으면 회사, 직무, 현재 문항 기준으로 필요한 경험을 스스로 고른다.
사용자가 확정한 `companies/<company-name>/company.md`가 있고 현재 문항과 유사한 질문 해석이 정리돼 있으면, 합격 자소서 레퍼런스보다 먼저 읽고 기준으로 삼는다.
합격 자소서 레퍼런스에서는 경험 내용 자체보다 문항에 답하는 방식, 문단 구성, 가독성, 논리 연결, 설득 흐름을 우선적으로 읽는다.
타회사 합격 자소서는 현재 회사와 문항이 충분히 유사할 때만 참고하고, 회사 이해나 지원동기 근거로 직접 사용하지 않는다.

## 톤 결정

1. 사용자가 원하는 톤을 직접 지정하면 그것을 우선한다.
2. 별도 지정이 없으면 해당 회사의 합격 자소서 레퍼런스를 기준으로 톤을 추정한다.
3. 합격 자소서가 없으면 회사, 직무, 문항 성격을 기준으로 톤을 결정한다.

## 문항별 작업

1. 각 문항은 `companies/<company-name>/questions/q1.md`, `q2.md` 같은 개별 파일에서 관리한다.
2. 각 문항 파일에는 최소한 다음 내용이 포함되어야 한다.
   - 참조 자산
   - 질문 태그
   - 유사 문항 참조
   - 문항 원문
   - 글자 수 제한
   - 문항 의도 분석
   - 경험 배치 이유
   - 공고 키워드 매핑
   - 사용할 경험과 근거
   - 초안
   - 리뷰어 평가
   - 수정 반영
   - 최종 답변
3. `## 참조 자산`에는 실제로 읽은 파일만 적는다. 각 줄은 `- 경로 | 이 문항에서 어떻게 썼는지` 형식으로 남긴다.
4. `## 참조 자산`에는 최소한 `assets/resume.md`, `assets/portfolio.md`, `assets/credentials.md`, `companies/<company-name>/company.md`, `companies/<company-name>/references/` 아래 파일 1개 이상이 포함되어야 한다.
5. 현재 회사 폴더에 기존 질문 파일이나 `before`, `after`, `comparison` 같은 작업 이력이 있으면 `companies/<company-name>/questions/` 아래 파일도 최소 1개 이상 포함해야 한다. 공통 과거 지원서보다 현재 회사 작업물을 먼저 본다.
6. `## 질문 태그`에는 `question-library/README.md`의 분류 체계를 따라 `대표 질문 유형:`과 `보조 태그:`를 적는다.
7. `대표 질문 유형`은 `motivation`, `job-fit-and-career-vision`, `problem-solving`, `collaboration`, `communication-and-conflict`, `strengths-and-weaknesses`, `achievement-and-challenge`, `education-and-learning`, `values-and-criteria`, `adaptability-and-attitude` 중 하나를 사용한다.
8. `## 유사 문항 참조`에는 `companies/*/questions/*.md` 아래 기존 문항을 최소 1개 이상 적는다. 각 줄은 `- 경로 | 대표 질문 유형과 보조 태그 기준으로 왜 유사한지, 무엇을 가져올지` 형식으로 남긴다.
9. 유사 문항이 있으면 현재 회사 여부와 관계없이 그 문항을 먼저 읽고, 표현 톤, 경험 배치, 질문 대응 방식을 현재 답변의 시작점으로 삼는다.
10. 한 문항의 작성 루프는 기본적으로 `writer pass -> reviewer pass -> revision pass -> final validation` 순서를 따른다.
10. writer pass에서는 현재 문항의 초안만 작성한다. 이 단계에서는 스스로 평가하거나 변호하지 않는다.
11. reviewer pass에서는 초안을 떨어뜨릴 이유를 찾는 감점식 평가만 수행한다. 이 단계에서는 문장을 다시 쓰지 않는다.
12. reviewer pass 입력에는 질문 원문, 글자 수 제한, 회사와 직무 핵심 요약, 초안, 평가 체크리스트를 우선 사용한다. 초안에 없는 사실은 추정하지 않는다.
13. reviewer는 초안의 좋은 점보다 `질문 미충족`, `회사/직무 연결 부족`, `경험 나열형`, `해석 부족`, `추상 표현 과다`, `AI 같은 흐름`을 먼저 찾는다.
14. reviewer 결과는 최소한 `리뷰어 총평`, `리뷰어 반복 횟수`, `핵심 보완 포인트`, `수정 반영 요약`, `최종 판정`, `잔여 미흡 항목`, `회사 레퍼런스 구조 비교` 형식으로 남긴다.
15. `## 리뷰어 평가` 섹션은 reviewer 결과를 그대로 남기는 자리로 사용한다. 위 항목이 모두 없으면 reviewer pass를 수행하지 않은 것으로 본다.
16. revision pass에서는 reviewer가 지적한 문제를 중심으로만 수정한다. 새 경험 추가는 최소화하고 핵심 메시지는 유지한다.
17. reviewer와 revision의 내부 내용 루프는 기본적으로 최대 2회까지만 반복한다.
18. 초안이 경험 나열형이면 먼저 경험 수를 줄이고, 각 경험 뒤의 해석과 배운점을 보강한다.
19. 한 문항 안에 핵심 경험이 2개를 넘으면 기본적으로 재배치가 필요한 상태로 본다.
20. 현재 회사 자료가 부족하면 문항이 같은 타회사 합격 자소서를 reviewer 비교용 보조 기준으로 쓸 수 있다.
21. 현재 회사의 합격 자소서와 구조 비교가 필요하면 reviewer 단계에서 먼저 수행한다. 비교할 항목은 문단 순서, 주장 제시 위치, 근거 배치 방식, 마무리 방식이다.
22. 최종 확정본으로 넘길 때는 reviewer의 `최종 판정`을 `통과`로 두고, reviewer 반복 횟수는 기본적으로 `2회`를 적는다. 1회만으로 끝낸 초안은 제출 후보가 아니라 중간본으로 취급한다.
23. `잔여 미흡 항목`에는 reviewer 기준에서 아직 남아 있는 약점을 적고, 최종 확정본에서는 `없음`으로 닫는다.
24. `회사 레퍼런스 구조 비교`에는 비교 결과를 한 줄로 요약하거나, 현재 회사 레퍼런스 부재처럼 비교가 불가능한 구체적 사유를 적는다.
25. validation은 내부 내용 루프가 끝난 뒤 마지막에만 수행한다.
26. validation은 형식과 표현 습관을 검사하는 마지막 게이트다. validation 통과만으로 제출 가능한 답변이라고 보지 않는다.
27. validation 실패 시에는 원칙적으로 형식과 표현만 수정한다. validation 수정을 위해 내용 의미가 크게 바뀌면 reviewer를 한 번 더 거친다.
28. CLI에는 내부 루프를 숨기지 말고 현재 문항의 `초안`, `리뷰어 평가`, `수정 반영`, `최종 답변`을 순서대로 보여준다.
29. 사용자에게 보여줄 때는 최소한 `질문 원문`, `현재 공백 포함 글자 수`, `문항 글자 수 제한`, `초안`, `리뷰어 총평`, `수정 후 답변`을 함께 제시한다.
30. reviewer 단계는 내부 메모로만 감추지 않는다. 사용자가 원하면 reviewer 총평과 핵심 보완 포인트를 그대로 확인할 수 있어야 한다.
31. 사용자가 현재 문항을 완료했다고 판단한 경우에만 다음 문항 파일을 작성한다.
32. 초안은 시작점으로만 사용하고, 제출 전에는 반드시 개인 경험과 회사 맥락을 반영해 다시 쓴다.
33. 합격 자소서에서 좋은 전개 방식을 찾더라도, 해당 지원자의 경험과 성과를 현재 지원자의 것처럼 재사용하지 않는다.
34. 문항별로 공고 키워드는 1개에서 2개 정도만 자연스럽게 녹이고, 나열식으로 붙이지 않는다.

## 평가와 수정

1. 초안을 작성한 뒤에는 validation부터 돌리지 말고 reviewer 평가를 먼저 수행한다.
2. reviewer 평가는 `references/evaluation.md`의 통과 기준과 미흡 신호를 따른다.
3. reviewer가 찾은 문제는 `references/revision.md`의 수정 원칙에 맞춰 revision pass에서 반영한다.
4. reviewer 결과에서 미흡 항목이 남아 있으면 최대 2회까지 내부 내용 루프를 반복한다.
5. reviewer 루프가 끝난 뒤에만 `scripts/validation.py`를 수행한다.
6. `scripts/validation.py`는 각 문항 파일의 `## 글자 수 제한`과 `## 최종 답변` 섹션을 함께 검사해야 한다.
7. `scripts/validation.py`는 `## 참조 자산` 섹션에 필수 자산과 실제 파일 경로가 모두 있는지, `## 유사 문항 참조` 섹션에 기존 질문 파일 경로가 있는지, `## 리뷰어 평가` 섹션에 reviewer 형식 필수 항목이 모두 있는지, 최종 판정이 `통과`인지, 반복 횟수가 `2회`인지, 잔여 미흡 항목이 `없음`인지도 함께 검사한다.
8. 해석과 배운점이 약하면 validation 전에 무조건 다시 쓴다.
9. validation에 실패하면 실패 원인을 반영해 형식과 표현을 먼저 수정한다.
10. 검증을 통과한 뒤에는 현재 문항의 초안, 리뷰어 평가, 수정 반영, 최종 답변을 CLI에 보여주고 사용자 피드백을 받는다.
11. 사용자 피드백이 있으면 현재 문항 파일만 다시 수정하고 필요 시 reviewer 평가와 final validation을 다시 거친다.
12. 최종본 확정 전에는 레퍼런스 문장과 지나치게 비슷한 표현이 남아 있지 않은지 다시 확인한다.
13. 내용 평가, reviewer 루프, 스타일 검증, 사용자 확인이 모두 끝난 뒤에만 현재 문항을 완료 처리한다.
14. 사용자 수정본이 최종본으로 확정되면 해당 수정이 왜 더 나은지 짧게라도 남겨 다음 문항과 다음 세션의 기준으로 재사용한다.
15. 외부 시스템에 최종 결과를 기록할 때는 각 문항별로 최소한 `질문 원문`, `글자 수 제한`, `최종 답변`을 함께 저장한다.
